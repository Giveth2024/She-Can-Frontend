 2. Comprehensive Use Case Analysis
 Use Case 1: Public User Submits Contact Form
 Actor: Guest OR Public Web Visitor
 Pre-conditions: The visitor has navigated to the base URL (`/`) of the web application.
 Trigger: The visitor fills in the fields and clicks the "Submit" button.
 Main Flow:
    1. System displays the form containing Name, Email, and Message fields.
    2. User inputs their data.
    3. User clicks "Submit".
    4. Client-side validation triggers to check for basic correctness (non-empty fields, structured email format).
    5. Frontend displays a loading spinner and disables the submit button to prevent double-submitting.
    6. Frontend sends an HTTP `POST` request to the Express API backend.
    7. Backend validates the payload.
    8. Backend commits records to the Aiven MySQL database.
    9. Backend sends back an HTTP `201 Created` status code.
    10. Frontend captures success response, stops the loading spinner, clears out the form input fields, and displays: “Form Submitted Successfully”.
 Alternative Flows (Error Handling):
 Validation Fails: If data is malformed, frontend halts request and shows inline red errors. If backend validation fails, it throws a `400 Bad Request`, and the frontend catches the error to show a user-friendly alert message.
 Server/Network Down: If Express or MySQL is unreachable, the frontend intercepts the network failure and presents an error message (e.g., "Server is temporarily unreachable. Please try again later."), keeping the user's typed input intact.

 Use Case 2: Admin Authentication, Audit Logging & Access Control
 Actor: Foundation Administrator
 Pre-conditions: Admin tries to navigate directly to the dashboard URL (/admin/dashboard) or accesses the specific production link (e.g., [https://your-app.vercel.app/admin]).
 Trigger: Attempting to access a protected administration route.
 Main Flow:
1. Next.js middleware hooks into Clerk to check for an active user session token.
2. If no valid session is found, Clerk intercepts the request and cleanly redirects the user to the Login page (/sign-in).
3. The Admin enters their valid foundation credentials into the Clerk interface.
4. Clerk authenticates the user credentials and generates a secure session token.
5. Instead of going straight to the dashboard, Clerk redirects the admin to an intermediate Loading/Sync page (/admin/auth-callback).
6. While the admin sees a loading state, the frontend dispatches a background POST request to the Express backend (e.g., /api/admin/log-signin), carrying the Clerk token.
7. The Express backend validates the token, extracts the admin's details, and executes an INSERT/UPDATE query into an admin_logs table in the Aiven MySQL database to track the exact login timestamp and user information.
8. The backend returns a successful 200 OK or 201 Created response back to the frontend.
9. Upon receiving this successful backend response, the Next.js frontend triggers a final redirect, sending the admin cleanly onto the actual Admin Dashboard (/admin/dashboard).

 Use Case 3: Admin Manages and Deletes Form Submissions
 Actor: Authenticated Foundation Administrator
 Pre-conditions: User has successfully completed the Use Case 2 authentication and log-sync flow, and is now actively sitting on the Admin Dashboard (/admin/dashboard).
 Trigger: The dashboard component mounts or the admin interacts with a "Delete" option next to a specific record.
 Main Flow:
1. The Next.js dashboard requests the list of records from the backend via GET /api/submissions, passing the validated Clerk JWT/Bearer token inside the HTTP Authorization header.
2. The Express backend interceptor intercepts the incoming request and validates the Clerk token to verify identity.
3. The backend queries the Aiven MySQL database using: SELECT id, name, email, message, created_at FROM submissions ORDER BY created_at DESC;
4. MySQL returns the dataset, and the backend sends it back to the Next.js frontend as a structured JSON array.
5. The Admin panel captures the JSON array and displays the records cleanly to the admin user in a responsive grid layout.
6. The Admin reviews the submissions and clicks the "Delete" action button next to an unwanted record.
7. The frontend dispatches a secure DELETE /api/submissions/:id request to the backend, carrying the Clerk JWT in the headers and the targeted record ID in the URL parameter.
8. The backend authenticates the token, drops the record from MySQL via: DELETE FROM submissions WHERE id = ?; and replies with a 200 OK success message.
9. The frontend UI catches the 200 OK status code and instantly re-fetches the list or filters the local state to remove the deleted row from the user's view without requiring a full page refresh.

 3. Strict Acceptance Criteria (AC)
 AC 1: Public Form Component
 Must visually format cleanly across mobile viewports (375px) up to desktop monitors (920px) via Tailwind utility utilities.
 Must dynamically show a processing/loading state while waiting on backend fulfillment; the submit button must be disabled during this time.
 Must render exact phrase “Form Submitted Successfully” upon successful receipt of a `201` HTTP response status.

 AC 2: Form & Input Validation Constraints
 Name Field: Must be trimmed string, minimum length of 2 characters, maximum of 125 characters. Cannot be null or blank spaces.
 Email Field: Must match standard RFC 5322 regex validation pattern (e.g., `user@domain.com`).
 Message Field: Must be a trimmed string, minimum length of 10 characters, maximum of 2500 characters.
 Fail-Safe: Validation must exist both client-side (for UX) and server-side (for security).

 AC 3: Admin Dashboard Secure State
 Any unauthenticated request attempting to hit `/admin` must be actively dropped and redirected to sign-in instantly by Clerk.
 The backend database credentials must never be exposed to the frontend browser context. All database operations must go strictly through the Express wrapper.


4. Full Backend API Specification & Testing Plan
ENDPOINT 1 
 Method: POST
 Endpoint: /api/submissions
 Access Level: Public
 Payload Requirement: JSON body containing { name, email, message }
 Success Response: 201 Created
 Body: { success: true, message: "Form Submitted Successfully" }

 Expected Failure Responses:
 400 Bad Request (Validation errors such as empty fields or invalid email format)
 500 Internal Error (Database connection down or server-side failure)

ENDPOINT 2 
 Method: POST
 Endpoint: /api/admin/log-signin
 Access Level: Protected (Clerk Auth Token Required)
 Payload Requirement: JSON body containing admin metadata or extracted via token
 Success Response: 200 OK or 201 Created
 Body: { success: true, message: "Login event logged successfully" }

 Expected Failure Responses:
 401 Unauthorized (Missing, expired, or invalid Clerk token)
 500 Internal Error (Failed to write to admin_logs table in Aiven MySQL)

 ENDPOINT 3 
 Method: GET
 Endpoint: /api/submissions
 Access Level: Protected (Clerk Auth Token Required)
 Payload Requirement: None (Bearer Token passed dynamically in the HTTP Authorization Header)
 Success Response: 200 OK
 Body: An array of objects [ { id, name, email, message, created_at }, ... ]

 Expected Failure Responses:
 401 Unauthorized (Missing or invalid Clerk session token)
 500 Internal Error (Database query failure or connection drop)

 ENDPOINT 4 
 Method: DELETE
 Endpoint: /api/submissions/:id
 Access Level: Protected (Clerk Auth Token Required)
 Payload Requirement: Route parameter "id" passed as an integer in the URL path
 Success Response: 200 OK
 Body: { success: true, message: "Deleted" }

 Expected Failure Responses:
 401 Unauthorized (Missing or invalid Clerk session token)
 404 Not Found (The targeted submission ID does not exist in the database)
 500 Internal Error (Database failure during deletion command execution)

 COMPLETE BACKEND TESTING PLAN (MANUAL & INTEGRATION ROUTING) 
1. Public Submission Unit Test:
 Action: Dispatch POST to /api/submissions with complete, valid JSON schema fields.
 Verification: Assert response status code is exactly 201. Verify using a database GUI client that a new row matching the payload has been committed to the submissions table with an accurate system timestamp.
2. Malformed Payload Validation Test:
 Action: Dispatch POST to /api/submissions with an invalid email address format, empty message field, or missing parameters entirely.
 Verification: Assert response code is exactly 400 Bad Request. The JSON response body must explicitly return error parameters outlining which field failed validation rules.
3. Unauthenticated Data Leaks Protection Test:
 Action: Attempt to issue a direct GET request to /api/submissions without passing any Authorization header or session tokens.
 Verification: Assert response returns HTTP code 401 Unauthorized. The application layer must completely block database access and query execution.
4. Session Tracking Verification Test:
 Action: Emulate a successful Clerk sign-in by hitting /api/admin/log-signin with a valid auth token.
 Verification: Assert response code is 200 OK/201 Created. Check the admin_logs table to confirm a new row correctly captures the timestamp of the access event.
5. Malicious ID Deletion Integrity Test:
 Action: Pass a non-existent, negative, or string-formatted ID parameter to the delete route (e.g., DELETE /api/submissions/999999 or /api/submissions/abc).
 Verification: Ensure the Express server handles the structural parsing cleanly without crashing, returning either a logical 404 Not Found or a handled 400 Bad Request error.

2. DATABASE ARCHITECTURE DESIGN (AIVEN MYSQL)
Table 1 Name: submissions
    Description: Stores incoming public contact form submissions.
    Columns:
        id: INT, Primary Key, Auto-Increment, Not Null.
        name: VARCHAR(100), Not Null (Trimmed string).
        email: VARCHAR(255), Not Null (Validated email string).
        message: TEXT, Not Null (Sanitized user text block).
        created_at: TIMESTAMP, Default CURRENT_TIMESTAMP, Not Null.

Table 2 Name: admin_logs
    Description: Tracks backend authorization events and logs when foundation administrators access the panel.
    Columns:
        id: INT, Primary Key, Auto-Increment, Not Null.
        clerk_user_id: VARCHAR(150), Not Null (Unique string signature from Clerk).
        admin_email: VARCHAR(255), Not Null (Email of the logged-in administrator).
        action_performed: VARCHAR(100), Not Null (e.g., "LOGIN_VERIFICATION", "RECORD_DELETION").
        ip_address: VARCHAR(45), Nullable (For basic system auditing security).
        logged_at: TIMESTAMP, Default CURRENT_TIMESTAMP, Not Null.
